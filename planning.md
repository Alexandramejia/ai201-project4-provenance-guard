# Provenance Guard Planning

## Milestone 1: System Understanding and Architecture

Provenance Guard is a backend system that checks submitted writing and estimates whether it is likely generated by AI, likely written by a human, or uncertain. A creator submits text through `POST /submit`. The system runs the text through two detection signals: a Groq LLM classification signal and a stylometric heuristics signal. The two scores get combined into one confidence score. Based on that score, the system creates a transparency label, saves the decision in the audit log, and returns a JSON response to the user.

If a creator disagrees with the result, they can submit an appeal through `POST /appeal`. The appeal includes the original `content_id` and the creator's reasoning. The system updates the content status to `under_review`, saves the appeal in the audit log, and returns a confirmation response.

## Detection Signals

### Signal 1: Groq LLM Classification

This signal uses Groq to judge whether the text appears generated by AI or written by a human. It looks at overall tone, structure, coherence, and style. This is useful because AI generated writing often sounds polished, balanced, and predictable.

Blind spot: this signal may wrongly flag polished human writing, formal academic writing, or writing from a non native English speaker as AI generated.

### Signal 2: Stylometric Heuristics

This signal uses Python text statistics such as sentence length variation, vocabulary diversity, and punctuation patterns. It captures structural writing patterns instead of meaning. This is useful because human writing tends to vary more in sentence length and word choice, while AI generated writing tends to be more uniform.

Blind spot: this signal may struggle with very short writing, poetry, heavily edited human writing, or simple repetitive writing.

## False Positive Scenario

A false positive happens when a real human writer gets labeled as AI generated. This is harmful because it can unfairly question the creator's originality. To reduce this risk, the system will use an uncertain label for middle confidence scores instead of forcing a yes or no result. If the creator disagrees, they can submit an appeal explaining why the work is original, and the content status will change to `under_review`.

## API Surface

### POST /submit

Accepts:

```json
{
  "text": "The submitted writing goes here.",
  "creator_id": "user123"
}
```

Returns:

```json
{
  "content_id": "abc123",
  "attribution_result": "uncertain",
  "confidence_score": 0.54,
  "label": "We're not confident enough to say whether this was written by AI or a human.",
  "signals": {
    "groq_score": 0.61,
    "stylometric_score": 0.47
  },
  "status": "reviewed"
}
```

The `attribution_result` field is the raw classification (`likely_ai`, `likely_human`, or `uncertain`), and `label` is the plain language transparency text a reader would actually see on the platform.

### POST /appeal

Accepts:

```json
{
  "content_id": "abc123",
  "reason": "This is my own writing. I just edit heavily and use a formal tone."
}
```

Returns:

```json
{
  "content_id": "abc123",
  "status": "under_review",
  "message": "Your appeal has been recorded and will be reviewed."
}
```

### GET /log

Lets anyone reviewing the system see the audit trail directly, so decisions and appeals don't just live in the README.

Returns:

```json
{
  "entries": [
    {
      "content_id": "abc123",
      "creator_id": "user123",
      "attribution_result": "uncertain",
      "confidence_score": 0.54,
      "signals": {
        "groq_score": 0.61,
        "stylometric_score": 0.47
      },
      "label": "We're not confident enough to say whether this was written by AI or a human.",
      "status": "reviewed",
      "appeal": null
    }
  ]
}
```
